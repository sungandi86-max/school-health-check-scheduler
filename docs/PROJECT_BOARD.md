# School Health Hub Project Board Guide

이 문서는 GitHub Project에서 School Health Hub 작업을 관리하는 기본 방식을 정리합니다.

## 컬럼 구조

### Backlog

아직 Sprint에 포함하지 않은 아이디어와 장기 과제를 둡니다.

예시:

- 학교별 관리자 권한 설계
- 보건실 방문 관리 모듈 아이디어
- AI 운영 보고서 자동 작성 검토
- 교육청 통계 확장 가능성 조사

### Todo

이번 또는 다음 Sprint에서 진행하기로 결정한 작업을 둡니다. 요구사항이 비교적 명확해야 합니다.

예시:

- 운영 분석 빈 상태 안내 추가
- 학교 설정 Supabase 테이블 설계
- 교사용 현황판 모바일 QA
- README 문서 링크 정리

### In Progress

현재 구현 또는 문서 작업이 진행 중인 항목을 둡니다.

예시:

- Sprint 25 v3.0 문서 정리
- 운영 보고서 인쇄 스타일 QA
- Supabase Realtime 재연결 처리 점검

### Review

구현은 끝났지만 빌드, 문구, 사용자 흐름, 코드 리뷰가 필요한 항목을 둡니다.

예시:

- `npm run build` 결과 확인 필요
- 교사용 화면 개인정보 노출 여부 검토
- 운영센터 문구 현장 용어 확인
- README 링크 동작 확인

### Done

완료 조건을 만족하고 커밋/푸시까지 끝난 항목을 둡니다.

예시:

- Sprint 21 역할 기반 화면 안내
- Sprint 22 학교 기본 설정
- Sprint 23 인쇄용 운영 보고서
- Sprint 25 v3.0 릴리스 노트 초안

## 카드 작성 규칙

각 Project 카드에는 다음 정보를 포함합니다.

- 목표
- 작업 범위
- 완료 조건
- 관련 파일 또는 화면
- 추천 커밋 메시지
- 필요한 경우 참고 스크린샷 또는 현장 메모

## 운영 원칙

- Backlog는 아이디어 저장소로 사용하고, Todo부터 실제 약속된 작업으로 봅니다.
- In Progress 항목은 동시에 너무 많이 만들지 않습니다.
- Review에서는 기능 동작뿐 아니라 개인정보 노출과 현장 문구를 함께 봅니다.
- Done으로 옮기기 전 빌드 통과와 커밋 여부를 확인합니다.
