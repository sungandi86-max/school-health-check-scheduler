# Testing Guide

School Health Hub의 테스트 환경과 작성 기준을 정리한 문서입니다.

## 테스트 도구

- Vitest: Vite 기반 단위 테스트 실행기
- React Testing Library: React 컴포넌트 렌더링 및 사용자 관점 테스트
- @testing-library/jest-dom: DOM matcher 확장
- jsdom: 브라우저 환경 모의 실행

## 테스트 실행 방법

전체 테스트를 1회 실행합니다.

```bash
npm run test
```

파일 변경을 감시하며 테스트를 반복 실행합니다.

```bash
npm run test:watch
```

배포 전에는 빌드와 테스트를 함께 확인합니다.

```bash
npm run build
npm run test
```

## 테스트 작성 기준

- 기능 코드와 가까운 위치에 `*.test.ts` 또는 `*.test.tsx` 파일을 둡니다.
- 컴포넌트 테스트는 사용자에게 보이는 동작을 중심으로 작성합니다.
- 유틸 함수 테스트는 입력과 출력이 분명한 계산 로직을 우선합니다.
- localStorage를 사용하는 테스트는 테스트 간 데이터가 섞이지 않도록 초기화합니다.
- Supabase 연결이 필요한 코드는 실제 네트워크 호출 없이 fallback 또는 순수 로직을 우선 검증합니다.

## 주요 테스트 대상

우선순위가 높은 테스트 대상은 다음과 같습니다.

- 검사 유형 선택과 첫 진입 흐름
- 세션 생성, 수정, 선택 유틸
- 명렬표 업로드 후 학생 상태 요약
- 운영센터의 현재/다음/완료/미도착 상태 계산
- 교사용 공유 링크와 리로스쿨 발송문 생성
- 운영 보고서 요약 계산
- localStorage fallback 동작

## 현재 기본 테스트

현재 포함된 기본 테스트는 다음을 검증합니다.

- `HealthCheckTypeSelector` 렌더링 및 선택 콜백
- 세션 저장 키, 세션 생성, 세션 수정
- 운영 요약, 학급 완료/미도착 상태, 학생 상태 요약
- 교사용/관리자/보고서 URL 및 리로스쿨 공유 문구 생성

## Codex 작업 후 테스트 확인 절차

Codex가 기능을 수정한 뒤에는 다음 순서로 확인합니다.

1. 변경 범위를 확인합니다.
2. 관련 테스트를 추가하거나 기존 테스트를 수정합니다.
3. `npm run test`를 실행합니다.
4. `npm run build`를 실행합니다.
5. 실패가 있으면 원인을 수정한 뒤 다시 실행합니다.
6. 테스트로 확인하지 못한 부분은 수동 확인 항목으로 작업 완료 보고에 남깁니다.

## 테스트 확장 계획

다음 Sprint에서 확장할 수 있는 테스트는 다음과 같습니다.

- Supabase Repository mock 테스트
- Realtime 구독 해제 테스트
- 운영 보고서 인쇄용 데이터 구성 테스트
- 파일 업로드 parser 테스트
- 스케줄러 자동배정 회귀 테스트
